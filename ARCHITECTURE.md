# ARCHITECTURE.md — Homelab Bot

## Суть проекта

Telegram-бот на Rust, живущий в приватном чате с единственным разрешённым пользователем. Использует Forum Topics (Threaded Mode) вместо отдельного webapp — вся функциональность идёт через топики внутри одного чата с ботом.

## Конфигурация

### .env (секреты, не коммитится)

- `BOT_TOKEN` — токен бота
- `CHAT_ID` — ID приватного чата
- `NOTIFY_SERVER_PORT` — порт HTTP-сервера для приёма внешних нотификаций
- `ROOT` — абсолютный путь для хранения быстрых заметок (`.md` файлы)
- `DEBOUNCE_SECS` — таймер дебаунса в секундах (по умолчанию 45)

### config.json (коммитится)

- `notifications` — thread_id топика "Уведомления"
- `quick_notes` — thread_id топика "Быстрые заметки"

Заполняется автоматически при первом запуске. Если файл отсутствует или пуст — бот создаёт топики через Bot API и записывает полученные `thread_id` в `config.json`.

## Архитектура запуска

Два независимых tokio-задачи в одном процессе:

```
main.rs
├── task: Bot (teloxide dispatcher)
│   └── слушает входящие в "Быстрых заметках"
│
└── task: HTTP-сервер (axum)
    └── ловит нотификации с локалки → шлёт в "Уведомления"
```

Общее состояние — `Arc`:
- `Arc<Bot>` — оба используют для отправки сообщений
- `Arc<Config>` — оба читают `thread_id`

Каждая задача обёрнута в `run_supervised` (`src/supervisor.rs`), который ловит панику через `catch_unwind` и рестартует с задержкой в 1 секунду. Запуск через `tokio::join!` — оба супервизора работают вечно, `main` не завершается. Падение одного таска не роняет другой. `systemd Restart=on-failure` остаётся страховкой на случай падения процесса целиком (OOM, kernel panic).

## Модули

### 1. Быстрые заметки

Топик "Быстрые заметки". Пользователь пишет текстовые сообщения прямо в этот топик, бот слушает входящие сообщения, отфильтрованные по `message_thread_id`, и сохраняет в `ROOT` как `.md` файлы.

#### Константы

```rust
const TITLE_MAX_LEN: usize = 50;
```

#### Состояние

Один активный буфер на топик. Хранится в `Arc<Mutex<Option<InboxBuffer>>>`:

```rust
struct InboxBuffer {
    file_path: PathBuf,
    close_handle: JoinHandle<()>,
    feedback_msg_id: MessageId,
}
```

#### Логика на каждое входящее сообщение

Сначала определяется, является ли сообщение **явным заголовком** (title marker): первая строка текста начинается с `!`. Если да — всё, что идёт после `!` на этой же строке (до конца строки), становится заголовком; остаток сообщения (если после первой строки есть текст) идёт в тело файла как обычно. Символ `!` в сам файл не попадает.

**Ветки:**

1. **Нет `!`-маркера И буфер пуст (окно неактивно):**
   - Имя файла = `{timestamp}.md` (только дата/время).
   - Контент записывается целиком (включая первую строку).

2. **Нет `!`-маркера И буфер уже активен (окно продлевается):**
   - Дописать в конец текущего файла целиком, продлить таймер.

3. **Есть `!`-маркер И буфер пуст (окно неактивно):**
   - Заголовок = текст после `!` на первой строке.
   - Имя файла = `{timestamp}_{slug}.md`.
   - Первая строка (с `!`) пропускается при записи в файл.
   - Остаток сообщения (строки после первой, если есть) — в тело файла.

4. **Есть `!`-маркер И буфер уже активен (окно продлевается):**
   - **Принудительное закрытие текущего окна** (сброс `close_handle`, обнуление `Option<InboxBuffer>`) без ожидания истечения таймера.
   - Заголовок = текст после `!`.
   - Создание **нового** файла.
   - Первая строка пропускается, остаток — в тело нового файла.
   - Запуск нового окна на `DEBOUNCE_SECS`.

Смысл: `!`-маркер — это ручной override дебаунс-таймера и способ задать осмысленное имя файлу. Без маркера бот просто группирует сообщения по времени, не пытаясь угадать тему. С маркером — создаёт файл с читаемым именем и принудительно начинает новую группу.

Если сообщение состоит только из `!` без текста после — это невалидный маркер (пустой заголовок). Поведение — как будто маркера не было (дата/время в имени, контент целиком); не паниковать, не игнорировать сообщение молча.

#### Запись на диск

Запись происходит **сразу**, на каждое сообщение, а не по таймеру. Таймер отвечает только за то, продолжает ли следующее сообщение писаться в этот же файл или откроет новый. Это даёт устойчивость к падению процесса — максимум что теряется при рестарте это "какой файл сейчас активен", не сами данные.

При наличии `!`-маркера первая строка пропускается при записи в файл (она уже в имени файла). Без маркера записывается весь текст целиком.

#### slugify()

- Кириллица остаётся как есть
- Пробелы → `-`
- Вырезаются: `/ \ : * ? " < > | #` и управляющие символы
- Множественные `-` схлопываются, ведущие/хвостовые обрезаются

```rust
fn slugify(s: &str) -> String {
    let cleaned: String = s
        .chars()
        .filter(|c| !"/\\:*?\"<>|#".contains(*c))
        .collect();
    let slug = cleaned
        .split_whitespace()
        .collect::<Vec<_>>()
        .join("-");
    slug.trim_matches('-').to_string()
}
```

#### first_line() + extract_explicit_title()

```rust
fn first_line(text: &str) -> &str {
    text.lines().next().unwrap_or(text)
}

fn extract_explicit_title(text: &str) -> Option<&str> {
    let first = text.lines().next().unwrap_or(text);
    let rest = first.strip_prefix('!')?;
    let trimmed = rest.trim();
    if trimmed.is_empty() { None } else { Some(trimmed) }
}
```

`extract_explicit_title` возвращает `Some(заголовок)` если первая строка начинается с `!` и после него остаётся непустой текст. `None` — если маркера нет, либо после `!` только пробелы.

#### truncate_at_word_boundary()

```rust
fn truncate_at_word_boundary(s: &str, max_len: usize) -> &str {
    if s.chars().count() <= max_len {
        return s;
    }
    let end = s.char_indices()
        .nth(max_len)
        .map(|(i, _)| i)
        .unwrap_or(s.len());
    let truncated = &s[..end];
    truncated.rfind(' ').map_or(truncated, |i| &truncated[..i])
}
```

Версия с исправленным off-by-one (старая через `take(max_len).last()` теряла на один символ больше лимита) и с обязательно сохранённой ранней проверкой `chars().count() <= max_len`. Без этой проверки функция режет по последнему пробелу даже строки, которые изначально короче лимита.

#### Формат файла

Без `!`-маркера:
```markdown
---
created: {ISO8601 timestamp}
source: telegram
---

{полный текст сообщения}
```

С `!`-маркером:
```markdown
---
created: {ISO8601 timestamp}
source: telegram
---

{текст сообщения без первой строки (с !)}
```

**Пример без `!`-маркера** — три сообщения подряд, одно окно:

Сообщения:
```
Надо пересмотреть debounce окно для inbox бота
Может 45 сек мало, буду печатать по частям и рвать мысль
Или наоборот слишком много и будет объединять разные темы
```

Имя файла: `2026-07-04_21-03.md`

```markdown
---
created: 2026-07-04T21:03:11+03:00
source: telegram
---

Надо пересмотреть debounce окно для inbox бота

Может 45 сек мало, буду печатать по частям и рвать мысль

Или наоборот слишком много и будет объединять разные темы
```

**Пример с `!`-маркером** — пользователь начинает новую тему:

Сообщение:
```
! Сайт-визитка
Нужен лендинг для проекта
```

Имя файла: `2026-07-04_21-15_Сайт-визитка.md`

```markdown
---
created: 2026-07-04T21:15:42+03:00
source: telegram
---

Нужен лендинг для проекта
```

#### Фильтрация

Обработчик подписывается только на `MessageKind::Text`. Нетекстовые сообщения (фото/голос/файлы) молча игнорируются.

#### Проверка ROOT

Бот НЕ создаёт `ROOT` директорию. Наличие `ROOT` — инфраструктурная ответственность окружения. Молчаливое создание скрывает реальную проблему конфигурации. При старте — явная проверка, fail fast:

```rust
if !root.is_dir() {
    panic!("ROOT path {:?} does not exist or is not a directory. \
            Create it manually before starting the bot.", root);
}
```

С `systemd Restart=on-failure` это даёт понятный сигнал (процесс не поднимается, причина видна в логах) вместо тихого расхождения данных.

### 2. Уведомления

Топик "Уведомления". Только вывод от бота в чат, пользователь туда не пишет.

Источник — независимый существующий Python-скрипт бэкапа (не трогаем). Канал доставки — внутренний HTTP на порту `NOTIFY_SERVER_PORT`.
